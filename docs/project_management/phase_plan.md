# 多类型应用/游戏自动截图采集平台阶段计划

## 阶段总览

本项目按 P0 到 P12 分阶段推进。P0 只建立项目文档、治理协议和架构基线；后续阶段必须在对应计划明确、测试策略明确、架构约束满足后再实施。

| 阶段 | 名称 | 目标 | 交付边界 |
| --- | --- | --- | --- |
| P0 | 项目基线与开发治理 | 固化项目文档、阶段计划、Codex 工作协议、核心架构规则 | 只写文档，不写业务代码，不接模型，不改 UI，不新增依赖 |
| P1 | MVP 基础框架 | 建立任务、状态、配置、存储元数据、Worker 抽象的最小框架 | 可运行基础骨架，核心规则测试覆盖 |
| P2 | 本地暂存、上传确认、清理流 | 建立本地暂存、百度网盘人工确认、删除保护与保留文件机制 | 上传前不删除，确认后按规则清理 |
| P3 | 模型网关 | 建立低频 AI 决策网关，支持可替换开源/本地模型优先策略 | 不让 AI 做高频逐帧操作 |
| P4 | 多类型 Worker 与行为包 | 建立浏览器、普通软件、Android、PC 游戏 Worker 与行为包机制 | PC 游戏 high 桶走行为包 + OBS/FFmpeg 抽帧 |
| P5 | 补采机制与人工补种子 | 支持未达标任务补采、人工补种子、失败恢复与审计 | 不突破禁止操作边界 |
| P6 | 环境配置与模型部署预备 | 建立单机开发、四机生产、机器角色、环境变量、模型 manifest 和检查脚本 | 只做配置和校验，不下载模型、不启动服务 |
| P7 | 四机并发与生产化压测 | 支持四机并发、资源隔离、生产化压测与运行看板 | 验证吞吐、稳定性、数据完整性 |
| P12 | 行为包自我深化引擎 | 基于历史运行结果、失败样本、人工补种子和质量反馈优化行为包策略 | AI 只参与低频分析与建议 |

## 阶段准入规则

- 每个阶段开始前必须确认阶段目标、范围、测试策略和验收标准。
- 任何业务代码变更必须有对应核心逻辑测试。
- 不允许跳过 P0 架构基线直接实现 Worker、模型或上传清理逻辑。
- 架构规则冲突时先停下说明，由架构师确认。

## 全局架构约束

- 截图桶只有 fixed、low、high；fixed 可选。
- low 或 high 至少出现一种。
- valid_total 必须 >= 1000 才能进入 capture_completed。
- valid_total 必须 <= 5000。
- capture_completed 后进入 upload_pending。
- 用户确认已上传百度网盘后，才能进入 uploaded_confirmed。
- 只有 uploaded_confirmed 后，才允许删除本地图片和临时视频。
- 删除后必须保留 summary.json、meta.jsonl、upload_manifest.json、run.log。
- PC 游戏 high 桶必须使用行为包 + OBS/FFmpeg 抽帧。
- AI 只做启动、场景判断、按钮定位、卡住恢复等低频决策。
- 不允许自动处理验证码、支付、充值、购买、聊天发送、账号安全验证、反作弊绕过。
- app-screenshot-agent 只作为 Android Worker 和公共质量模块的复用基础，不整体替代新平台架构。
