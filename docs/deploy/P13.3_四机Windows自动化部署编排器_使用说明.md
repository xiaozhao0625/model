# P13.3 四机 Windows 自动化部署编排器使用说明

P13.3 的目标是把四台 Windows 机器的部署过程产品化为可检查、可编排、可审计、可汇总的脚本体系。它不是最终生产部署完成证明，也不要求 W1/W2/W3 已真实接入。

## 本阶段不做什么

- 不下载大模型。
- 不安装 PaddleOCR 或 EasyOCR。
- 不处理账号登录、验证码、支付、反作弊、聊天或账号安全验证。
- 不把 `.env` 原文、数据库密码、API key、token 或 credentials 写入日志和 Git。
- 不让 W1/W2/W3 直接连接 PostgreSQL，worker 只通过 M0 Master API 通信。

## 目录说明

- `deploy/software_catalog.json`: 统一软件资产清单。
- `deploy/role_matrix.json`: M0/W1/W2/W3 的角色、工具、服务、端口和模块矩阵。
- `deploy/roles/*.yaml`: 面向人工阅读的角色清单。
- `scripts/p13/*.ps1`: P13.3 的可执行编排脚本。
- `scripts/p13/lib/*.ps1`: 检测、日志、网络、GPU、服务、安装、安全 JSON 工具库。
- `logs/p13/<ROLE>/`: 每台机器的本地结果目录，禁止提交。
- `deploy_output/`: M0 汇总报告目录，禁止提交。

## M0 首次操作

```powershell
cd E:\work\model
.\scripts\p13\00_prepare_ssh.ps1 -Role M0
.\scripts\p13\01_inventory_local.ps1 -Role M0
.\scripts\p13\02_install_role.ps1 -Role M0 -Mode Plan
```

所有真实安装动作都必须先运行 `Plan`，确认 `logs/p13/M0/install_plan.json` 后再运行 `Execute`。

## W1/W2/W3 首次 bootstrap

在每台 worker 的管理员 PowerShell 中按 `logs/p13/ssh/ssh_bootstrap_steps.md` 执行 OpenSSH Server bootstrap。只允许局域网内 M0 到 W1/W2/W3 的受控连接，不要求暴露外网访问。

## 运行 Inventory

```powershell
.\scripts\p13\01_inventory_local.ps1 -Role M0
.\scripts\p13\01_inventory_local.ps1 -Role W1
.\scripts\p13\01_inventory_local.ps1 -Role W2
.\scripts\p13\01_inventory_local.ps1 -Role W3
```

结果写入 `logs/p13/<ROLE>/inventory.json`、`missing_tools.json`、`version_report.json`、`network_report.json`、`gpu_report.json`、`service_report.json` 和 `inventory_summary.md`。

## 运行 PlanInstall 和 Execute Install

```powershell
.\scripts\p13\02_install_role.ps1 -Role W1 -Mode Plan
.\scripts\p13\02_install_role.ps1 -Role W1 -Mode Execute
```

`Plan` 只生成计划，不安装软件。`Execute` 会对可用 `winget_id` 的工具尝试 winget 安装；OBS、Android Emulator、PostgreSQL、Redis、NVIDIA Driver 等复杂项可以输出 `manual_install_required`。

## 启动服务

```powershell
.\scripts\p13\03_start_role_services.ps1 -Role M0
.\scripts\p13\03_start_role_services.ps1 -Role W1
```

脚本复用已有 `scripts/deploy/p13/*.bat` 启动入口，并输出 operator action。它不会读取或记录 `.env` 原文。

## 健康检查

```powershell
.\scripts\p13\04_health_check_role.ps1 -Role M0
.\scripts\p13\04_health_check_role.ps1 -Role W2
```

每个检查项都会包含 `status`、`message`、`next_action`。端口不通输出 `port_unreachable`，远端不可达输出 `remote_unreachable`，worker 未注册输出 `worker_not_registered`。

## 收集日志

```powershell
.\scripts\p13\05_collect_logs.ps1 -Role W3
```

压缩包会排除 `.env`、credentials、截图、视频、模型文件、安装包、`node_modules`、虚拟环境和已有 zip。

## M0 编排

```powershell
.\scripts\p13\m0_orchestrator.ps1 -Action Inventory
.\scripts\p13\m0_orchestrator.ps1 -Action PlanInstall
.\scripts\p13\m0_orchestrator.ps1 -Action Install -Role W1
.\scripts\p13\m0_orchestrator.ps1 -Action StartServices
.\scripts\p13\m0_orchestrator.ps1 -Action HealthCheck
.\scripts\p13\m0_orchestrator.ps1 -Action CollectLogs
.\scripts\p13\m0_orchestrator.ps1 -Action FullDryRun
```

汇总报告写入 `deploy_output/overall_deploy_report.json` 和 `deploy_output/overall_deploy_report.md`。阅读报告时先看每个角色的 `Status`、`Missing / failed` 和 `Next action`，再进入对应的 `logs/p13/<ROLE>/` 查看细节。

## 常见失败原因

- OpenSSH Server 未安装或防火墙未放行，远端记录为 `remote_unreachable`。
- 服务未启动或端口未监听，健康检查记录为 `port_unreachable`。
- worker runtime 未启动或 Master API 不可达，记录为 `worker_not_registered`。
- `winget` 不可用，Execute 安装步骤记录为 `failed` 并给出下一步建议。
- NVIDIA Driver、PostgreSQL、Redis、Android Emulator 需要人工安装或人工初始化。

## 安全注意事项

不要提交 `logs/`、`deploy_output/`、`.env`、`.env.*`、credentials、diagnostics zip、截图目录、安装包、模型文件、`node_modules`、虚拟环境或缓存目录。所有真实安装动作都必须先 `Plan` 后 `Execute`。
